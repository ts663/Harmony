  ---
  Architecture Compliance

  Gap: status field is not in the architecture document.
  The unified-backend-architecture.md User entity (В§3.2 class diagram) and ERD
  (В§4.2 users table) do not include a status field. The issue explicitly
  requires it, so it's a valid addition вҖ” but the arch doc should be updated to
  reflect this schema change.

  Gap: getUser ignores publicProfile flag.
  The arch doc (line 1436) states: "Users with public_profile = false are 
  displayed as 'Anonymous' with no avatar." The current getUser returns the full
   record regardless. An authenticated user can call user.getUser({ userId: 
  "..." }) to retrieve the displayName, avatarUrl, and username of any user who
  has opted out of a public profile. This is a privacy bug.

  Minor: Repository layer skipped.
  The arch doc defines a UserRepository with findById and getPublicProfile
  (В§3.7). The PR skips this in favor of direct Prisma calls, which is consistent
   with how channel.service.ts is implemented. This is acceptable given the
  established project pattern, but worth noting.

  ---
  Code Issues

  user.router.ts вҖ” Hardcoded enum values in UserStatusSchema

  // user.router.ts:5
  const UserStatusSchema = z.enum(['ONLINE', 'IDLE', 'DND', 'OFFLINE']);

  This duplicates the Prisma UserStatus enum as string literals. If the enum is
  ever modified, this schema won't fail at compile time. Use
  z.nativeEnum(UserStatus) instead:

  import { UserStatus } from '@prisma/client';
  const UserStatusSchema = z.nativeEnum(UserStatus);

  user.service.ts вҖ” getCurrentUser is redundant

  // user.service.ts:22-24
  async getCurrentUser(userId: string) {
    return userService.getUser(userId);
  },

  This is a pure pass-through with no added logic. It adds a layer of
  indirection with no benefit. Either give it distinct behavior (e.g., always
  return full profile), or merge it into the router directly. As written, it
  creates a false impression of differentiated behavior.

  Migration вҖ” drops and re-adds indexes across two migrations

  20260306223212_add_user_status/migration.sql drops three canonical DESC
  indexes that were defined outside Prisma's DSL, then
  20260306223300_fix_missing_indexes/migration.sql re-adds them. This is
  functional but leaves the DB in a degraded state between the two migrations
  running. On a fresh deploy this is instantaneous, but the approach is fragile.
   These drops should not have been auto-generated by Prisma вҖ” this suggests
  those indexes are not represented in the Prisma schema (correct for raw SQL
  indexes), and the migration was generated against a dirty local DB state. Not
  a blocking issue, but worth a comment.

  Tests

  - 12 tests covering all happy paths and NOT_FOUND cases вҖ” good.
  - The four status test cases (ONLINE, IDLE, DND, OFFLINE) run sequentially вҖ”
  OFFLINE is not explicitly tested (default only verified in getUser). Minor

  - No test for updateUser with an empty patch {} вҖ” should be a no-op and return
   existing data.

  the Prisma enum at the type level.

  Non-blocking:
  - Arch doc needs a status field added to the User model definition.
  - getCurrentUser passthrough is unnecessary.
  - Missing OFFLINE status test and empty-patch test.
