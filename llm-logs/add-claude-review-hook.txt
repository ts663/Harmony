     Plan: Automated Claude PR Review Workflow

     Context

     Set up a GitHub Actions workflow that automatically reviews every
     new/updated PR using anthropics/claude-code-action. The review should
     submit a formal GitHub review (approve or request changes) covering code
     quality, security, project conventions, and test coverage.

     Approach

     1. Add GitHub Secret

     In the repo's GitHub Settings ‚Üí Secrets and variables ‚Üí Actions, add:
     - ANTHROPIC_API_KEY ‚ÄĒ your Anthropic API key

     GITHUB_TOKEN is provided automatically by Actions.

     2. Create Workflow File

     File: .github/workflows/claude-pr-review.yml

     name: Claude PR Review

     on:
       pull_request:
         types: [opened, synchronize]
         branches: [main]

     jobs:
       review:
         runs-on: ubuntu-latest
         permissions:
           contents: read
           pull-requests: write
         steps:
           - uses: anthropics/claude-code-action@beta
             with:
               anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}
               github_token: ${{ secrets.GITHUB_TOKEN }}
               direct_prompt: |
                 You are reviewing a pull request for the Harmony project ‚ÄĒ a 
     search-engine-indexable
                 chat application built with Next.js 16 (frontend) and Express +
      tRPC 11 (backend),
                 using TypeScript 5, Prisma 5 ORM, PostgreSQL 16, and Redis.

                 Review this PR thoroughly against the following criteria:

                 **Code Quality & Style**
                 - TypeScript strict mode; no implicit `any`; unused vars/params
      must be prefixed with `_`
                 - Prettier formatting: 100-char line width, semicolons, single
     quotes, trailing commas
                 - ESLint 9 rules must be respected
                 - Backend: camelCase for code, snake_case in DB (mapped via
     Prisma `@map()`)
                 - Frontend: React components in PascalCase, hooks prefixed
     `use`

                 **Security**
                 - JWT tokens must not be stored insecurely or leaked in logs
                 - Auth middleware must be applied to all protected tRPC/REST
     routes
                 - No raw SQL that could lead to injection ‚ÄĒ use Prisma query
     builder
                 - No secrets/credentials hardcoded

                 **Project Conventions**
                 - REST routes are for public/unauthenticated access (SEO
     crawlers); tRPC is for
                   authenticated admin operations ‚ÄĒ do not mix these
                 - Prisma `ChannelVisibility` enum values (PUBLIC_INDEXABLE,
     PUBLIC_NO_INDEX, PRIVATE)
                   must not be renamed or altered
                 - New DB fields must follow snake_case naming and be added to
     `schema.prisma`
                 - Keep Section 3 (Class Diagram) and Section 4 (List of
     Classes) in docs in sync
                   if architectural changes are made

                 **Test Coverage**
                 - Backend changes must include or update Jest tests in
     `tests/**/*.test.ts`
                 - Tests must be able to run against a real PostgreSQL + Redis
     instance
                 - Do not leave test files that import non-existent modules

                 After your review, submit a formal GitHub PR review using the
     `gh` CLI:
                 - Use `gh pr review <PR_NUMBER> --approve -b "<summary>"` if
     the PR is ready to merge
                 - Use `gh pr review <PR_NUMBER> --request-changes -b "<detailed
      feedback>"` if changes are needed
                 - Include specific file:line references in your feedback
                 - Do NOT leave nitpick-only comments; focus on issues that
     matter

     3. Workflow Behavior

     - Triggers on every PR opened or updated targeting main
     - Claude checks out the PR diff via the action's built-in GitHub context
     - Claude posts a formal review (approve or request changes) ‚ÄĒ not just a
     comment
     - Runs on ubuntu-latest with only contents: read + pull-requests: write
     permissions (least privilege)

     Files to Create

     - .github/workflows/claude-pr-review.yml (new file)

     Files to Modify

     None.

     Verification

     1. Open a test PR targeting main ‚ÄĒ the Claude PR Review workflow should
     appear in the Checks section
     2. Verify the workflow completes and a formal GitHub review
     (approve/request changes) appears on the PR
     3. Push a new commit to the open PR ‚ÄĒ verify the workflow re-runs
     (synchronize event)
     4. Check Actions logs to confirm the ANTHROPIC_API_KEY secret is being
     consumed (no 401 errors)

     Notes

     - The anthropics/claude-code-action@beta tag is the current stable release
     channel for this action
     - If the repo has branch protection rules requiring reviews, this bot
     review can satisfy that requirement ‚ÄĒ configure "Required approving
     reviews" to accept reviews from GitHub Actions
     - The action automatically handles diff fetching and GitHub context
     injection; no manual git diff step is needed
